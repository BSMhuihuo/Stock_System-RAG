# 执行摘要

本报告围绕“股票系统”项目需求，深入分析了所需的岗位技能、系统模块、RAG/Agent设计、大模型训练、合规风控、技术选型、项目计划及面试演示准备等方面。项目目标是构建一个集数据采集、存储、特征工程、预测模型、推荐、交易执行、告警、监控、界面与合规于一体的智能股票交易系统，并通过该项目展示 Agent、RAG 和大模型相关技能。报告首先列出各岗位（ML 工程师、NLP/LLM 工程师、数据工程师、后端/DevOps、量化研究员、产品/项目经理）所需技能、工具和经验，并给出常见面试题型和作品示例要点。其次，逐个模块详细定义子功能、输入输出接口、性能指标、数据频率和保留策略。随后讨论检索增强生成（RAG）与智能体（Agent）的设计，包括向量数据库选型、检索策略、提示工程、链式调用和多 Agent 协作等；同时给出典型架构流程图。接着介绍大模型及训练策略，包括开源与 API 模型候选、微调方法、数据构建和评估指标（如回测收益、预测精度、夏普比率、最大回撤等【30†L85-L93】【37†L60-L62】）。在数据安全与合规方面，强调敏感数据处理、交易合规审计日志、异常回滚及沙箱模拟交易等要求，并援引监管规定如证监会“程序化交易管理规定”对维护交易秩序的指导【34†L5-L11】。技术选型部分通过 mermaid 架构图展示系统组件与数据流，并给出开源库与商业服务（向量数据库、LLM、云服务等）的优缺点对比表，估算自托管与云服务的成本范围。项目实施计划按 3、6、12 个月分阶段规划交付物、验收标准和面试演示材料（如 README、演示视频、可运行代码等）。最后，报告列出面试与演示准备要点，包括常见问题示范回答、代码 walkthrough、可展示的 Demo 和回测示例，以及如何解释 RAG/Agent 设计权衡。全文穿插中文权威资料和官方文档引用，深入评估可行性、风险与成本，并给出推荐的实施路径与最小可行产品（MVP）清单。

## 项目需求与功能清单

### 数据采集（Data Ingestion）

- **子功能：**行情数据（股票价格、指数、成交量等）采集；财务数据（财报、指标）抓取；新闻及舆情（研报、公告、新闻、社交媒体）爬取；用户行为数据（模拟或历史交易）记录；外部信号（宏观数据、行业报告）。  
- **输入接口：**通过第三方API（如同花顺、雪球、Alpha Vantage、聚源）、Web爬虫、RSS/新闻订阅；调度系统（Crontab、Airflow）定时触发数据抓取。  
- **输出接口：**原始数据存入数据湖/消息队列（Kafka/MinIO/HDFS）；结构化数据入库（数据库、Parquet 文件）。可提供实时订阅接口（WebSocket/RESTful API）供系统其他模块消费。  
- **性能指标：**行情数据延迟要求极高（秒级或毫秒级）；新闻抓取可容忍批量（分钟级）；吞吐量要求稳定支持海量数据，例如每秒数千条行情推送。  
- **数据频率与保留：**行情数据为秒级/分级，通过时序数据库或列式存储长期保留（通常至少1-3年）；财报、公告类数据每日或月度获取，保留多年；非结构化文本原文及关键词提取结果保留。历史原始数据需备份，采用冷热分层存储，定期归档或淘汰（例如7年政策留存要求）。  

### 数据存储与治理（Storage & Governance）

- **子功能：**构建数据仓库/数据湖，进行数据清洗和标准化（日期格式、证券代码对齐）；元数据管理（表结构、字段解释）；数据一致性校验与监控；访问权限控制与审计；备份恢复策略。  
- **输入接口：**由数据采集模块流入各类数据库：时序数据库（InfluxDB/ClickHouse）存行情；关系型数据库（PostgreSQL/MySQL）存用户/订单数据；NoSQL（MongoDB/Redis）缓存热数据；数据仓库（Hive、Parquet 存储）用于离线分析。  
- **输出接口：**对外提供查询接口（SQL API、OLAP 报表）；调度系统按需触发数据质量检查任务；支持数据科学平台（Jupyter/Spark）访问。  
- **性能指标：**查询响应时延依数据类型而定：交互式查询≤200ms（OLAP索引优化）；ETL处理批作业可数小时完成；实时插入/更新需高并发（每秒千条以上）。  
- **数据频率与保留：**交易相关数据和关键信息长期保留（≥7年），遵循金融数据监管要求；热数据（最近1年）高性能存储，冷数据归档。数据治理策略包含定期清洗（缺失值填充）、变更追踪和数据血缘记录，确保可追溯和合规审计。

### 特征工程

- **子功能：**从原始行情中提取技术指标（如移动平均、成交量比、技术指标 RSI、MACD 等）；生成因子（基本面因子、财务比率、Sentiment 指标）；时间序列特征（滞后值、差分）；文本特征（基于自然语言处理提取情感分、关键词）；特征归一化/降维（PCA、t-SNE等）。  
- **输入接口：**调用数据仓库和数据湖中的原始数据；接收参数配置（如窗口大小、因子公式）。  
- **输出接口：**生成特征表或特征向量存入特征存储层（如专用特征数据库或Redis缓存），供模型训练和预测调用。  
- **性能指标：**批量特征计算采用分布式计算（Spark/Flink），要求能在每日结束时计算完毕前一天数据（数小时内完成）；少量需要实时计算的指标（如分钟级走势）须接近实时响应。  
- **数据频率与保留：**特征数据随着原始数据更新（分钟级行情可分钟更新）；对历史特征做缓存，长期保留以支持历史回测和模型训练。版本化管理特征生成逻辑，方便回滚和复现。

### 预测模型

- **子功能：**股票价格/涨跌预测（回归或分类模型）、组合回报预测；风险预测（波动率、极值）；场景模拟（蒙特卡洛模拟）。常用模型包括：时间序列模型（ARIMA、LSTM、Transformers）、机器学习模型（XGBoost、随机森林）和深度强化学习。  
- **输入接口：**从特征存储获取特征向量；接受模型调用请求（如根据股票代码、日期输入返回预测结果）。  
- **输出接口：**生成预测值/信号，写入数据库或推送到事件总线；提供 RESTful API 供前端或其他服务查询。  
- **性能指标：**在线预测服务延迟要求在可接受范围（如<500ms）；批量训练可数小时完成。预测准确度和响应延迟需平衡，评估指标包括预测误差（MSE/RMSE）、分类准确率、F1、回测年化收益和夏普率等。  
- **数据频率与保留：**模型实时更新频率视策略而定，可日更或周更；预测结果与真实结果均需留存，便于事后评估和在线学习。保留最近若干年的预测日志与评估指标（如误差随时间变化）以检测模型漂移。

### 推荐系统

- **子功能：**基于模型输出和用户偏好，生成个性化股票推荐或交易策略。例如结合 RAG 检索结果（行业报告、历史案例）生成推荐理由；榜单推荐（趋势股、价值股等）。可构建协同过滤或基于内容的简单推荐，亦可用强化学习优化组合。  
- **输入接口：**用户画像和需求（可以是简要文本或表单）；模型预测结果与外部知识检索结果。  
- **输出接口：**推荐列表（带评分/理由），提供给前端界面或下单系统；生成交互式报告或可视化图表。  
- **性能指标：**交互式查询响应需快（<1s）以保证用户体验；推荐质量可通过离线用户点击率、组合收益来评估。  
- **数据频率与保留：**用户历史交互和点击数据实时更新，推荐模型可在线学习或定时重训练。保存用户反馈用于持续优化。保留用户操作日志和推荐结果供评估和合规审计。

### 交易执行

- **子功能：**自动化交易指令发送与管理：将决策系统生成的买卖信号转化为实际订单，路由到券商接口；风控检查（资金、持仓、交易限制等）；订单跟踪和成交确认；交易记录日志。  
- **输入接口：**来自推荐/信号系统的交易指令（股票代码、数量、价格、方向）；账户信息（资金余额、证券持仓）；配置的交易规则。  
- **输出接口：**券商下单 API 调用（如文华财经、大智慧、Interactive Brokers API）；交易状态通知（成交、失败等）推送到系统日志；账户持仓更新。  
- **性能指标：**非高频场景下，交易指令执行延迟可容忍在秒级；对延迟要求不如行情处理高，但需保证低失败率和高稳定性；并发能力需支持同时多笔委托。风控检查（如触发预设止损）需实时。  
- **数据频率与保留：**交易日志以毫秒时间戳记录每笔指令与反馈，长期保留以满足审计要求；资金与持仓记录实时更新并定期校验。

### 通知/告警

- **子功能：**预警监控（如股价临界点、交易异常、系统故障）以及正常推送（策略报告、收益通知）。包括邮件、短信、企业微信或APP推送接口。  
- **输入接口：**来自监控系统和业务系统的事件触发（阈值条件、日志异常、用户请求）；用户订阅的通知规则。  
- **输出接口：**多渠道发送接口（SMTP、SMS 服务、消息队列对接微信公众号/APP）；同时将告警信息写入告警日志和仪表板。  
- **性能指标：**对于关键告警需及时（通常秒级或分钟级推送）；高吞吐情况下保证告警不丢失（使用消息队列或ACK机制）。  
- **数据频率与保留：**告警事件记录、处理历程及确认结果需持久化，符合监管归档要求；一般保留至少一年。用户历史通知也应记录以便查询。

### 监控与审计

- **子功能：**系统运行监控（CPU/内存使用、延迟、错误率）；交易监控（每日P&L、风险敞口）、审计日志（所有用户操作、策略变更记录）。  
- **输入接口：**各模块输出的指标和日志（通过统计Agent或Sidecar导出指标）；数据库/消息队列日志；审计系统收集的操作记录。  
- **输出接口：**监控系统（如 Prometheus + Grafana、ELK）中展示仪表盘；异常告警触发到通知模块；合规报告生成（每日/每月报表）。  
- **性能指标：**指标采集频率通常 10s - 1min。监控系统可承受每秒数百条度量数据。告警机制应低误报、可配置阈值。审计日志写入需保证不可篡改，可定期备份。  
- **数据频率与保留：**监控数据保留短期可高分辨率（如保存最近90天细粒度，1年以上可降采样）；审计日志需要长期保存（例如根据监管需5年以上），并确保写时加签或上链以防篡改。

### 用户界面

- **子功能：**前端展示：行情图表、账户信息、交易窗口、策略报告；AI 交互界面（智能问答、报告生成）；图形可视化（K 线、指标曲线、组合表现）。  
- **输入接口：**用户输入（查询、交易指令）；后端数据接口（图表数据、推荐结果、交易反馈）。  
- **输出接口：**浏览器/客户端呈现（可用 React/Vue）；实时推送（WebSocket）更新数据；移动端/PC 兼容。  
- **性能指标：**前端请求响应 <200ms；图表加载优化；对于价格等需要近实时刷新；并发用户支撑视规模而定，可能需要负载均衡。  
- **数据频率与保留：**界面请求的数据缓存机制可减少后台负载；日志记录用户行为和页面异常（用于后续改进和审计）。

### 权限与合规

- **子功能：**基于角色的访问控制（RBAC）：不同用户（管理员、分析师、交易员、普通用户）权限隔离；敏感数据脱敏与加密存储；个人信息保护。  
- **输入接口：**身份认证模块（OAuth/OpenID，LDAP/SSO 集成）；权限服务查询（如 Auth0、Keycloak）。  
- **输出接口：**返回访问令牌（JWT 等），在后续请求中校验用户权限；生成审计日志条目（谁何时对哪个模块进行了何种操作）。  
- **性能指标：**权限检查一般开销极小（单次在毫秒级）；安全事件监控（暴力登录、异常访问）需实时检测。  
- **数据频率与保留：**审计日志（包括API调用、配置更改、交易操作）需长期保存并加密归档；符合证券交易等监管要求的合规日志存储（如《证券法》要求审计轨迹【34†L5-L11】）。

## RAG 与 Agent 设计

检索增强生成（Retrieval-Augmented Generation）与智能体（Agent）在本项目中扮演核心角色，用于增强决策的准确性和灵活性。以下重点考虑知识库与索引、向量数据库、检索与召回策略、提示工程、链式调用、多 Agent 协作、上下文管理等。

- **检索库与索引：**知识库类型包括结构化数据（数据库）、文档集（调研报告、公告）、代码片段、对话历史等。可结合**倒排索引**和**向量索引**。先使用关键词/全文检索（Elasticsearch/Solr）进行初步过滤，再用向量检索（如余弦相似度）精细排序。常见做法为先检索后排名（Rerank）【1†L45-L48】【37†L60-L62】。索引策略包括**分块（chunking）**长文档、摘要文档、设置文档记忆截断。定期重建索引以加入新数据，并记录每个文档来源以便审计。

- **向量数据库选型：**考虑 Qdrant、Milvus、Weaviate、Chroma、Elasticsearch 等。在**向量检索场景**下，Qdrant 适合 RAG 系统（支持在线更新、结构化过滤、多字段索引）【20†L49-L58】；Milvus 提供企业级分布式方案，适合海量向量（检索延迟<100ms，支持亿级向量，高并发）【22†L49-L57】【22†L88-L94】；Weaviate 强调混合搜索（向量+关键词）和 GraphQL 接口，但配置复杂【22†L74-L79】【22†L90-L97】；Chroma 轻量适用于开发调试，无原生持久化（依赖文件系统）【22†L66-L70】。根据需求：生产级 RAG 系统可选 Qdrant（易部署）或 Milvus（海量数据场景）；原型验证可用 Chroma；若使用云服务可考虑 Pinecone（商用、免运维，但成本较高）。下表列出常见选型对比：

- **检索器与召回：**检索策略包括**关键词检索**（基于默认搜索词）和**向量语义检索**结合，采用混合召回（Hybrid Retrieval）【1†L45-L48】。可使用句子嵌入（如 Sentence-BERT、ERNIE）构建向量检索器；支持多轮对话时保留上下文。检索结果可通过再排序策略（embedding 相似度、BM25）提高准确度。为降低延迟，可缓存热门问题结果，或使用增量检索（如基于最近查询调整向量）。

- **提示工程（Prompt Engineering）：**设计模块化 Prompt，覆盖各种场景，如行情分析、投资策略、风险提示等。支持模板化管理（例如使用 LangChain PromptTemplate【24†L9-L13】）。例如，对风险评估任务使用“作为专业顾问，请基于用户描述快速评估风险承受能力”类似格式（见反应式架构示例【15†L148-L156】）。需处理上下文长度限制，将重要信息摘要到 prompt 中；对 LLM 输出进行后处理规范格式。

- **链式调用与工具使用：**采用**LangChain 等框架**构建多个 Chain 和工具（tool）调用。每个 Agent 可以是一个多步骤链：先检索（Retriever），再生成回答（LLM），最后后处理和行动执行【24†L31-L40】【24†L57-L64】。工具包括实盘交易 API、计算引擎、特定规则函数等。链中可并行或串行调用模型和检索模块，结果汇总输出。

- **Agent 角色与决策流程：**系统采用多智能体架构：一个“主控Agent”拆解高层需求，多个“功能子Agent”分别负责子任务【38†L71-L80】。例如可有数据爬取Agent、策略生成Agent、交易Agent、风控Agent等。主控Agent 类似项目经理，协调子Agent 协作【38†L71-L80】。决策流程中，主控Agent根据用户指令或系统任务生成一系列子任务，交给对应Agent执行，其特点是“高内聚、低耦合”，每个子Agent独立开发、易扩展【38†L81-L90】。下图示意基于“主控+子Agent”的流程：

- **上下文管理与长期记忆：**每轮交互保留对话上下文，并可把重要信息写入长期记忆库（如用户偏好、策略学习结果）。长期记忆可存入向量数据库，用作未来检索提示，以减少遗忘。针对金融场景，可定期更新知识库（如定期导入新公告、研究报告），保证模型信息及时。需要综合考虑**成本与延迟权衡**：较大模型上下文可以提升准确性，但推理成本和延迟升高，故可使用异步查询或缓存机制降低响应时间【1†L45-L48】【24†L52-L60】。

## 大模型与模型训练

### 候选模型与微调

- **候选模型：**开源模型如 Baichuan、LLaMA 系列、Bloom、T5 等，以及国内模型（鹏程·盘古、海光等）；API 服务如 OpenAI GPT-4/3.5、Azure OpenAI、阿里文澜、百度 ERNIE Bot 等。选择时综合考虑模型规模、中文表现、金融领域知识。应用场景可优先考虑已微调金融领域的中文 LLM【38†L103-L111】。例如“幻方量化”自研模型或 Baichuan-13B 等对金融中文有不错效果；OpenAI 提供强大泛化能力但需英语提示并付费。  
- **微调策略：**根据需求采用**指令微调**（Instruction Tuning）或 LoRA/PEFT 精调技术，以注入金融领域知识和策略（用户可采用 HuggingFace PEFT 框架）。监督学习数据来源包括：历史交易日志、策略研究文档、专家生成的问答对等。可自行构建问答式或填空式训练集，例如按照实际策略问题提问、答案以交易建议形式给出。验证集要涵盖边界场景（极端行情、稀有事件）。  
- **多模态与数据增强：**虽然本项目主要为文本与数值任务，但可考虑扩展至图表（Candle图）、新闻图像。利用图像标注模型提取图表特征；对新闻或公告采用 NLP 提取情感。使用数据增强技术（时间序列抖动、新闻同义改写）提升模型鲁棒性。  

### 评估指标与监控

- **回测与交易绩效：**使用真实或模拟历史数据进行**策略回测**。关键指标包括累计收益、年化收益率、胜率、夏普比率、最大回撤、Calmar 比率等【30†L85-L93】【30†L94-L100】。推荐使用开源回测框架（如 Backtrader、Empyrical）生成绩效报告。所有交易信号、执行结果应记录以便回测复现。  
- **预测精度与风险指标：**针对预测模型，用均方误差（MSE）、平均绝对误差（MAE）、R²等评估回归精度；若做涨跌分类，用准确率、F1 值等。针对组合风险，计算监控指标如VaR、CVaR等。为减少过拟合，应对比训练集和验证集表现，使用交叉验证。  
- **在线学习与监控：**系统上线后持续监控模型表现：一旦预测偏差或回测收益显著下滑，需报警并触发模型重训练。可实现增量训练或周期性重新训练。在线学习需谨慎：建议在离线模式下先验证再逐步放量。使用 MLOps 工具（如 MLflow、Weights & Biases）记录实验结果和模型版本。  

## 数据安全、合规与风控

- **敏感数据处理：**严格按照《个人信息保护法》等要求，对用户和账户信息进行加密存储与访问控制。仅允许必要服务访问交易凭证和个人资料，记录访问日志；采用 TLS 等加密传输。定期进行安全审计和渗透测试。  
- **交易合规与审计日志：**遵循监管规定（如证监会“程序化交易管理规定”【34†L5-L11】），记录所有交易指令和变更操作。审计日志应包括用户ID、操作时间、操作内容，使用不可篡改的日志系统（如链式存储或写时签名）。系统需支持复盘和现场检查（例如按照监管要求提供可查询接口）。  
- **回滚与异常处理：**交易指令或系统异常时，必须有清晰回滚策略。比如若下单失败或资金不足，应自动取消挂单并通知用户；在数据管道错误时可回退至上一正常状态。所有异常（连接丢失、模型崩溃等）需触发告警并保留故障快照。使用事务日志或操作日志保证系统可恢复到任意检查点。  
- **模拟交易与沙箱：**开发初期和新策略上线前，应在历史数据或模拟环境中测试行为。构建沙箱环境（纸交易模式），用虚拟资金进行完整流程验证；使用策略风控测试，检查是否触发监管风险规则（如日内回转率异常、疑似内幕交易模式等）。模拟结果需与实盘环境隔离，防止误操作影响真实资金。
